# LLM-Powered Custom Policy Generation — Design Notes
> **SimuOrg Project** | Discussion Date: 2026-03-29

This document captures the full design discussion around using an LLM to generate custom
simulation policies for the SimuOrg engine. It covers the core problem, why naive approaches
fail, the correct architecture, and concrete examples using real calibration output.

---

## 1. The Feature Goal

Allow users to describe a workplace scenario in **natural language** and have the system
automatically produce a valid `SimulationConfig` that drives the simulation engine.

**Example user input:**
> *"Simulate an aggressive hiring freeze with heavy workload on remaining employees"*

**Expected output:**
```python
SimulationConfig(
    workload_multiplier=1.25,
    hiring_active=False,
    stress_gain_rate=2.8,
    motivation_decay_rate=0.015,
    shock_factor=0.25
)
```

---

## 2. The Core Problem: LLM Values Are Not Grounded in Real Data

### The Naive Approach (Wrong)

The first instinct is to let the LLM directly output absolute `SimulationConfig` values.

**Why this fails:**

The LLM has no knowledge of the uploaded dataset. The same natural language prompt would
produce the same numbers for every company — but different companies have fundamentally
different organizational dynamics.

**Example:**
```
User prompt: "Simulate an aggressive hiring freeze"

LLM outputs:  workload_multiplier=1.25, stress_gain_rate=2.8

Company A:    already stressed org (16% attrition) → these values cause catastrophic collapse
Company B:    calm org (8% attrition) → these same values barely register
Company C:    near-burnout workforce → immediate total loss of headcount
```

The LLM is essentially making up numbers that feel plausible from training data,
but are not grounded in any specific organization's reality.

---

## 3. Understanding the Two Layers in SimuOrg

Before designing the solution, it's critical to understand that SimuOrg operates on
**two separate layers** that are often confused:

### Layer 1 — Calibration Output (Absolute, Dataset-Specific)
These are the real, data-fitted engine rates stored in `calibration.json`.
They are derived from the uploaded employee dataset via the calibration pipeline.

```
stress_gain_rate        : 0.0102   ← fitted to your employees' actual stress dynamics
recovery_rate           : 0.0048
shockwave_stress_factor : 0.2489
annual_attrition_rate   : 0.1612   (16.12% — the historical ground truth)
prob_scale              : 5.025
monthly_natural_rate    : 0.0145
avg_burnout_limit       : 0.5902
```

**These values differ per company.** They are what make the simulation accurate for
a specific organization.

### Layer 2 — SimulationConfig (Relative, Policy-Level Multipliers)
These are the parameters in `policies.py` that describe a *scenario* on top of the
calibrated baseline.

```python
@dataclass
class SimulationConfig:
    workload_multiplier:    float = 1.0   # 1.0 = no change, 1.25 = 25% more workload
    motivation_decay_rate:  float = 0.005
    shock_factor:           float = 0.2
    hiring_active:          bool  = True
    layoff_ratio:           float = 0.0
    stress_gain_rate:       float = 1.0   # MULTIPLIER, not absolute rate
    overtime_bonus:         float = 0.0
    wlb_boost:              float = 0.0
```

> [!IMPORTANT]
> `SimulationConfig.stress_gain_rate` is a **multiplier** applied ON TOP of the
> calibrated `stress_gain_rate`. They are not the same quantity.
>
> ```
> Effective engine stress rate = calibrated_stress_gain_rate × SimulationConfig.stress_gain_rate
>                               = 0.0102 × 2.8   (hiring freeze scenario)
>                               = 0.02856
> ```

---

## 4. Why the "Delta Shift" Approach Also Has Limits

A follow-up idea was to have the LLM output *delta shifts* (relative changes) rather
than absolute values. For example: "increase stress by +1.8 from baseline" instead of
"set stress to 2.8".

**This is better, but still has problems:**

- The LLM still doesn't know whether a company is already stressed or calm
- A company near burnout limit needs a *smaller* delta to show the same scenario severity
- A calm company needs a *larger* delta to produce a noticeable effect
- The LLM still has no access to the calibration data when generating deltas

**Conclusion:** Even deltas cannot be accurately generated by an LLM in isolation,
because the appropriate magnitude depends on company-specific calibration context.

---

## 5. The Correct Answer: Why SimulationConfig Multipliers ARE Already Dataset-Safe

Here is the key insight that resolves the concern:

> **SimulationConfig values don't need to differ per company — because calibration
> already handles that at the engine level.**

```
Same SimulationConfig:  stress_gain_rate = 2.8 (hiring freeze multiplier)

Company A (stressed):   2.8 × 0.0102 = 0.0286  → 28% attrition over 6 months
Company B (calm):       2.8 × 0.0035 = 0.0098  → 11% attrition over 6 months
```

The **relative impact** of the policy is correctly preserved. A hiring freeze is a hiring
freeze — the severity as a *multiplier* is comparable across organizations. What differs
is the *absolute outcome*, and that's driven by the calibrated engine rates, not by
SimulationConfig.

The LLM does NOT need to generate different numbers for different companies for the
same described scenario.

## 6. The Clean Architecture: Three Cases

### Case A — Vague Natural Language (e.g., "it feels like a startup crunch")

**Flow:**
```
User natural language
        ↓
     [LLM]
        ↓
  Qualitative Profile             ← LLM is excellent at this
  {
    workload_level:  "very_high",
    hiring:          "active",
    stress_level:    "extreme",
    morale_impact:   "high"
  }
        ↓
  [Backend Mapping Table]         ← defined once using domain knowledge
        ↓
  SimulationConfig                ← dataset-agnostic multipliers
```

The LLM's job is **semantic parsing only** — understanding intent and mapping it to
your vocabulary. It never touches raw numbers.

**Mapping table example:**
```python
STRESS_LEVEL_MULTIPLIERS = {
    "low":      0.5,
    "moderate": 1.2,
    "high":     2.8,
    "extreme":  5.5,
}

WORKLOAD_MULTIPLIERS = {
    "normal":    1.0,
    "elevated":  1.1,
    "high":      1.2,
    "very_high": 1.25,
    "extreme":   1.45,
}
```

---

### Case B — Percentage-Based Instructions (e.g., "increase workload by 30%") ✅ Most Accurate

This is the **cleanest and most accurate** case. The user gives a precise instruction,
the LLM only parses structure, and the backend does pure arithmetic.

**Flow:**
```
User: "Increase workload by 30%"

LLM extracts:
{
  "changes": [
    {"param": "workload_multiplier", "operation": "multiply", "factor": 1.30},
    {"param": "stress_gain_rate",    "operation": "multiply", "factor": 1.15}
                                        ↑
                        LLM can also infer cascading effects
                        (more workload → more stress) with reasoning
  ],
  "reasoning": "30% more workload typically increases stress by ~15%"
}

Backend computes:
  baseline workload_multiplier = 1.0
  new value = 1.0 × 1.30 = 1.30   ← dead accurate, no guessing
```

**Why cascade inference is acceptable here:**
The cascade (workload → stress) is directionally correct even if imprecise. If the
user specified only workload, the LLM can flag the cascade as a suggestion:
*"I've also increased stress by 15% as a typical cascade. You can override this."*

---

### Case C — Named Scenario Matching (e.g., "simulate a layoff scenario")

The LLM maps the description to the closest named policy in `POLICIES` dict, then
optionally applies the user's additional modifiers on top.

```python
# LLM returns:
{
  "base_policy": "layoff",
  "overrides": {
    "layoff_ratio": 0.10   # user said "10% layoff" instead of default 15%
  }
}

# Backend:
config = get_policy("layoff")         # start from existing policy
config.layoff_ratio = 0.10            # apply user override
```

## 7. LLM Prompt Design

### For Percentage/Direct Instructions

```
System:
  You are a simulation parameter extractor for an organizational dynamics engine.
  The user will describe a workplace policy change.
  Extract ONLY which parameters change, and by what factor.

  Available parameters and their baseline values:
  - workload_multiplier    (baseline = 1.0,   range: 0.5 – 2.0)
  - stress_gain_rate       (baseline = 0.8,   range: 0.3 – 6.0)
  - motivation_decay_rate  (baseline = 0.005, range: 0.001 – 0.05)
  - shock_factor           (baseline = 0.0,   range: 0.0 – 0.8)
  - wlb_boost              (baseline = 0.0,   range: 0.0 – 1.0)
  - overtime_bonus         (baseline = 0.0,   range: 0.0 – 3.0)
  - layoff_ratio           (baseline = 0.0,   range: 0.0 – 0.3)
  - hiring_active          (boolean)

  Return valid JSON only. Do NOT invent values. If unsure, omit the parameter.

  {
    "changes": [
      {"param": "<name>", "operation": "multiply", "factor": <float>},
      {"param": "<name>", "operation": "set",      "value":  <value>}
    ],
    "reasoning": "<brief explanation>"
  }

User: "<user description here>"
```

### For Qualitative/Vague Descriptions

```
System:
  You are a workplace scenario classifier.
  Map the user's description to qualitative labels only.
  Do NOT output numbers.

  Return:
  {
    "workload_level":  "low" | "normal" | "elevated" | "high" | "very_high" | "extreme",
    "stress_level":    "low" | "moderate" | "high" | "extreme",
    "hiring":          "active" | "frozen",
    "morale_impact":   "positive" | "neutral" | "low" | "severe",
    "has_layoff":      true | false,
    "wlb_change":      "improved" | "unchanged" | "worsened",
    "closest_policy":  "baseline" | "remote_work" | "flexible_work" | "kpi_pressure" |
                       "hiring_freeze" | "layoff" | "promotion_freeze" | "overtime_pay" | "custom"
  }

User: "<user description here>"
```

## 8. Validation & Safety Layer (Always Required)

Since LLMs can hallucinate or produce out-of-range values, the backend must always
validate and clamp values before passing to `SimulationConfig`:

```python
PARAMETER_RANGES = {
    "workload_multiplier":   (0.5,   2.0),
    "stress_gain_rate":      (0.3,   6.0),
    "motivation_decay_rate": (0.001, 0.05),
    "shock_factor":          (0.0,   0.8),
    "wlb_boost":             (0.0,   1.0),
    "overtime_bonus":        (0.0,   3.0),
    "layoff_ratio":          (0.0,   0.3),
}

def clamp_param(name: str, value: float) -> float:
    lo, hi = PARAMETER_RANGES[name]
    return max(lo, min(hi, value))
```

> [!WARNING]
> Never pass raw LLM output directly to `SimulationConfig`.
> Always validate ranges and types before creating the config object.

## 10. Summary: Who Does What

| Responsibility | Handled By | Why |
|---|---|---|
| Understand natural language intent | LLM | Its primary strength |
| Output qualitative labels | LLM | Classification, not calculation |
| Parse "X% more workload" | LLM | Structural extraction only |
| Multiply baseline by extracted factor | Backend | Simple arithmetic |
| Validate and clamp output values | Backend | Safety, LLMs hallucinate |
| Set absolute engine rates per company | Calibration pipeline | Data-driven, per-dataset |
| Make scenarios feel right per company | Calibration layer | Already handled by design |

> [!NOTE]
> The LLM should **never** output raw SimulationConfig numbers directly.
> Its role is *translation* (natural language → structured intent), not *calculation*.
> The calibration engine ensures that structured intent produces accurate, company-specific outcomes.

---

## 11. Recommended Implementation Order

1. **Define the qualitative vocabulary** — finalize label sets for workload, stress, etc.
2. **Build the mapping table** — translate labels to SimulationConfig multiplier ranges
3. **Create the LLM extractor endpoint** — `/api/policy/generate` that calls LLM + returns config
4. **Add validation layer** — clamp all output, reject invalid params
5. **Build the % instruction path** — parse "increase X by Y%" → arithmetic
6. **Add scenario preview** — show user what the generated config will do before running simulation
7. **Optionally surface the reasoning** — show LLM's explanation for each parameter choice

## 12. The Final Architecture: Calibration Context + Industry Domain as LLM Input

### Why All Previous Approaches Were Still Incomplete

Every approach discussed so far — deltas, qualitative labels, domain formulas — shared one
common flaw: the LLM had **no knowledge of the specific company it was generating a policy for**.

The final, correct design feeds the LLM two additional inputs:

1. **Calibration output** — the company's actual fitted dynamics from `calibration.py`
2. **Industry / domain** — what sector the company operates in

This gives the LLM real context to reason from, making its output grounded in data rather
than generic assumptions.

---

### What Gets Passed to the LLM

#### Input 1 — Calibration Context (auto-generated, already exists)

```python
calibration_context = {
    "annual_attrition_rate":  0.1612,   # 16% is high → stressed org
    "avg_burnout_limit":      0.5902,   # employees near burnout at 59% capacity
    "stress_gain_rate":       0.0102,   # moderate stress accumulation
    "recovery_rate":          0.0048,   # slow recovery — interventions matter more here
    "avg_quit_prob":          0.1647,   # high baseline flight risk
    "wlb_stress_sensitivity": 1.118,    # WLB is a strong lever for this org
    "calib_quality":          "stable"
}
```

The LLM can use these to reason about severity:
- High `annual_attrition_rate` → org is already fragile, policies must be more impactful
- Low `recovery_rate` → benefits like rest days have above-average effect here
- High `wlb_stress_sensitivity` → WLB changes are more powerful than salary for this org

#### Input 2 — Industry / Domain

```
"Technology / Software"
```

Industry gives the LLM understanding of what motivates employees in this sector:
- **Tech** → autonomy, WLB, recognition > salary
- **Manufacturing** → safety, consistent pay, physical workload matters
- **Finance / Banking** → bonus culture, prestige, high stress tolerance
- **Healthcare** → purpose-driven, burnout is the dominant risk factor
- **Retail** → high turnover is already expected, salary is primary lever

#### Input 3 — User's Policy Description

```
"5% salary hike for everyone + 2 days off per month for next 3 months"
```

---

### How the LLM Reasons With Full Context

**System prompt structure:**

```
You are a simulation parameter advisor for organizational dynamics.

Company profile (from calibration data):
- Annual attrition: 16.12%    → stressed, high-turnover organization
- Avg burnout limit: 0.59     → employees operating near burnout capacity
- Recovery rate: 0.0048       → very slow recovery from stress
- WLB sensitivity: 1.118      → WLB changes have above-average impact here
- Quit probability: 0.1647    → high baseline flight risk

Industry: Technology / Software
Context: In tech, autonomy, WLB, and recognition typically outweigh salary.
         A 5% salary hike has moderate impact; extra time off has high impact
         for this sector. Given this org's slow recovery rate, rest interventions
         are especially effective.

Available SimulationConfig parameters (output changes only, not full config):
- workload_multiplier    (1.0 = baseline, range: 0.5–2.0)
- stress_gain_rate       (1.0 = baseline multiplier, range: 0.3–6.0)
- motivation_decay_rate  (0.005 = baseline, range: 0.001–0.05)
- wlb_boost              (0.0 = no boost, range: 0.0–1.0)
- overtime_bonus         (0.0 = no bonus, range: 0.0–3.0)
- shock_factor           (0.0 = no shock, range: 0.0–0.8)
- hiring_active          (boolean)
- duration_months        (integer, if policy is time-limited)

Return JSON only. Include a brief reasoning per parameter.

User policy: "5% salary hike for everyone + 2 days off per month for 3 months"
```

**LLM output (with context-aware reasoning):**

```json
{
  "config": {
    "overtime_bonus":        0.5,
    "motivation_decay_rate": 0.003,
    "wlb_boost":             0.35,
    "stress_gain_rate":      0.65,
    "duration_months":       3
  },
  "reasoning": {
    "overtime_bonus":        "5% salary hike → moderate financial lift. In tech, impact is below average vs equity/WLB, so conservative bonus increase.",
    "motivation_decay_rate": "Pay improvement slows motivation decay from 0.005 to 0.003. Effect amplified slightly due to high baseline quit probability (0.1647).",
    "wlb_boost":             "2 days off/month is significant. This org has high WLB sensitivity (1.118) and slow recovery (0.0048) — rest days will have above-average positive effect.",
    "stress_gain_rate":      "Extra rest days reduce stress accumulation. Lowered to 0.65 from baseline 0.8. Critical given this org's slow recovery rate.",
    "duration_months":       "Policy is time-limited to 3 months as stated by user."
  }
}
```

Notice the LLM correctly reasoned:
- WLB boost is stronger than salary for a tech org
- The high WLB sensitivity (1.118) amplifies the rest days further
- The slow recovery rate (0.0048) makes the rest intervention especially valuable for this company

This is **company-specific, industry-aware reasoning** — not a generic guess.

### Getting the Industry / Domain: Two Approaches

#### Option A — User selects from a dropdown (most reliable)

Present a simple selector on the upload page or simulation setup:

```
Industry / Domain:
  ○ Technology / Software
  ○ Manufacturing / Industrial
  ○ Banking / Finance
  ○ Healthcare / Pharma
  ○ Retail / E-commerce
  ○ Consulting / Professional Services
  ○ Education
  ○ Other (describe): ___________
```

#### Option B — Auto-detect from dataset columns (convenient, use as default)

The uploaded employee dataset often contains `job_role` or `department` columns.
The system can detect the likely industry from common values:

```python
INDUSTRY_SIGNALS = {
    "Technology / Software":   ["software engineer", "data scientist", "devops", "product manager", "backend", "frontend"],
    "Healthcare / Pharma":     ["nurse", "doctor", "pharmacist", "clinical", "lab technician"],
    "Banking / Finance":       ["analyst", "trader", "accountant", "compliance", "risk manager"],
    "Manufacturing":           ["operator", "technician", "quality control", "assembly", "production"],
    "Retail":                  ["sales associate", "store manager", "merchandiser", "cashier"],
}

def detect_industry(job_roles: list[str]) -> str:
    job_roles_lower = [r.lower() for r in job_roles]
    for industry, signals in INDUSTRY_SIGNALS.items():
        if any(signal in role for role in job_roles_lower for signal in signals):
            return industry
    return "Other"
```

**Best UX:** Auto-detect and show the result to the user for confirmation:
```
"We detected your organization is in: Technology / Software
 Is this correct? [Yes] [Change]"
```

### Why This Is the Correct Final Architecture

| Problem | How This Solves It |
|---|---|
| LLM doesn't know company stress level | Calibration profile tells it (attrition rate, burnout limit, recovery rate) |
| LLM doesn't know what motivates employees here | Industry context tells it (tech = WLB > salary) |
| Same policy means different things for different orgs | LLM scales its output based on sensitivity values (wlb_stress_sensitivity, etc.) |
| LLM making up numbers | It still outputs SimulationConfig multipliers (not absolute engine values), validated + clamped |

---

### Complete Final Data Flow

```
1. User uploads dataset
        ↓
2. System auto-detects industry from job_role column
   → Shows user: "Detected: Technology / Software — correct?"
        ↓
3. Calibration pipeline runs → calibration.json (per-company fitted values)
        ↓
4. User describes a custom policy in natural language
        ↓
5. Backend assembles LLM prompt:
   - Calibration profile (from calibration.json)
   - Industry context (from step 2)
   - Parameter definitions + valid ranges
   - User's policy description
        ↓
6. LLM responds with SimulationConfig changes + reasoning per parameter
        ↓
7. Backend validates + clamps all values to safe ranges
        ↓
8. SimulationConfig is constructed and passed to the simulation engine
        ↓
9. Engine uses calibration.json rates underneath → outcome is company-specific
        ↓
10. User sees simulation results + LLM's reasoning (optional, for transparency)
```

---

### Updated Implementation Order

1. **Calibration pipeline** — already complete ✅
2. **Industry detection** — auto-detect from `job_role` column + user confirmation UI
3. **Calibration profile builder** — extract the relevant subset of `calibration.json` for LLM context
4. **LLM prompt template** — build the system prompt with profile + industry + parameter definitions
5. **LLM API endpoint** — `POST /api/policy/generate` → calls LLM, returns config + reasoning
6. **Validation layer** — clamp all output values to PARAMETER_RANGES
7. **UI: Policy preview** — show user the generated config before running simulation
8. **UI: Reasoning display** — optionally show why the LLM chose each value
9. **Override controls** — let user manually adjust any parameter before running
