1. Executive Summary

1.1 Project Synopsis

SimuOrg is a full-stack, AI-powered workforce simulation platform that gives organizational leaders a quantitative, forward-looking view of employee attrition risk under different HR policy scenarios. By combining a calibrated XGBoost classifier with an agent-based simulation engine, a NetworkX-modelled organizational graph, and a natural-language LLM interface, SimuOrg transforms raw HR CSV data into actionable predictive intelligence.

The system enables a CEO or HR director to describe a policy in plain English — such as "implement a 15% salary raise across Engineering" — and receive a statistically validated, month-by-month forecast of attrition rate, headcount, stress, motivation, and productivity over a configurable simulation horizon.

2. Project Overview

2.1 Problem Statement

Traditional HR analytics are retrospective — they explain what happened, not what will happen under a new policy. When leadership decides to implement a hiring freeze, a promotion moratorium, or an overtime scheme, there is no scientific way to preview human capital consequences. Decisions default to intuition or incomplete benchmarks.

SimuOrg resolves this by turning actual employee data into a living simulation. Every individual is modelled as an autonomous agent with empirically-derived stress physics, burnout thresholds, loyalty dynamics, and a calibrated quit-probability from a trained gradient-boosted classifier.

2.2 Core Objectives

  • Train an XGBoost attrition classifier on uploaded HR data with automated feature engineering and probability calibration.
  • Implement an agent-based time-step simulation engine modelling stress accumulation, fatigue, motivation decay, and network contagion.
  • Integrate Llama 3.1 via Groq / local Ollama to translate natural-language CEO input into structured simulation configuration.
  • Provide a RAG layer via Pinecone to ground LLM outputs in real-world HR scenario precedents.
  • Expose all capabilities through a RESTful FastAPI backend with session-isolated multi-tenancy.
  • Offer per-employee SHAP explainability and a comprehensive model quality reporting API.

6. Machine Learning System Design

The ML core is a fully automated, session-scoped pipeline. Every CSV upload triggers a fresh training run as a FastAPI BackgroundTask, producing a calibrated XGBoost classifier whose per-employee attrition probabilities drive the simulation physics.

6.1 Model Architecture & Training Pipeline

Normalised Employee DataFrame
Feature Engineering — 12 derived features added
StratifiedKFold k=5 — split into train / calibration / validation
XGBoost.fit() — n_estimators=300, max_depth=6, scale_pos_weight=auto
Early Stopping — halts at round 20 if validation AUC stalls
IsotonicRegression — calibrates probabilities on held-out fold
Threshold Tuning — PR curve scan, argmax F1 threshold stored
Burnout Estimator — stress heuristic score per employee
SHAP TreeExplainer — exact attributions, global + per-employee
Binary-search Calibration — aligns simulated rate to historical ±0.5%
ModelStorage — persists model, threshold, alpha, SHAP values to DB
Hyperparameter Value Rationale
n_estimators 300 Balanced bias-variance for 500–5,000 row HR datasets
max_depth 6 Captures non-linear interactions without severe overfitting
learning_rate 0.1 Standard shrinkage; no grid search required at this scale
scale_pos_weight neg/pos ratio Compensates for typical 85:15 class imbalance in attrition data
eval_metric auc Threshold-agnostic ranking quality; works well with imbalanced labels

Raw HR Columns — salary, tenure, satisfaction, overtime…
income_ratio = monthly_salary ÷ department mean

overtime_stress = overtime_flag × workload_score

6.3 Calibration Pipeline

Raw XGBoost probabilities are systematically over-confident. SimuOrg applies isotonic regression and an empirical binary-search loop to align simulated attrition with historical ground truth.

XGBoost raw probabilities — over-confident
IsotonicRegression.fit() on calibration fold — monotone correction
Run 1-month pilot simulation with current scale factor alpha
|simulated_rate − historical_rate| ≤ 0.005?
Yes — converged
Store alpha + calibrated model
Typical: 8–15 iterations
No — adjust
Binary-search: update alpha
Max 50 iterations, then fallback alpha=1.0

6.4 Threshold Tuning

Calibrated probabilities on validation fold
Compute Precision-Recall curve — sklearn.metrics
Calculate F1 = 2·P·R / (P+R) at each threshold point
Select argmax(F1) — optimal_threshold stored in ModelStorage
At inference: P(attrition) ≥ threshold → flag as at-risk

Models Used

Model Provider Role Context Window Inference Speed
Llama 3.3 70B Versatile Groq Cloud (LPU) Reasoning Chain — complex policy logic & causality 8,192 tokens ~250–300 tokens/sec
Llama 3.1 8B Instant Groq Cloud (LPU) Intent Parsing — fast structured parameter extraction 8,192 tokens ~750+ tokens/sec
Llama 3.1 8B (GGUF Q4_K_M) Ollama (local CPU/GPU) Fallback — rate-limit & offline resilience 8,192 tokens ~30–50 tokens/sec
llama-text-embed-v2 Pinecone Inference Embedding — scenario vectorisation 512 tokens per chunk Serverless, managed

Model Architecture Properties

Property Value Notes
Architecture Transformer decoder (causal LM) Meta Llama 3 family
Parameters 70B (Reasoning) / 8B (Parsing/Fallback) Task-specific model routing
Tokenizer Llama 3 BPE tokenizer 128,000 vocabulary size
Quantization (Ollama) GGUF Q4_K_M (8B model) ~4.6 GB VRAM; runs efficiently on consumer hardware
Groq Execution LPU (Language Processing Unit) Enables 70B inference at interactive speeds
Temperature 0.15 – 0.2 Low value reduces creativity, guarantees strict JSON determinism

Why Dual-Model Routing (70B Reasoning + 8B Parsing) — Rationale

  • Reasoning Chain (70B via Groq): Complex HR policy scenarios require deep reasoning to understand causality and edge cases. The 70B model handles the heavy lifting in reasoning_chain.py, ensuring zero-hallucination JSON formatting and strict adherence to structural constraints.
  • Intent Parsing (8B via Groq): Simple intent mapping and parameter extraction does not require frontier reasoning. The 8B model handles these tasks in intent_parser.py, offering near-instant responses and saving API bandwidth.
  • Fallback (8B via Ollama): Running 70B locally is too slow and hardware-intensive (~40GB VRAM required). The 8B model provides a lightweight offline fallback for all tasks, ensuring 100% system uptime when Groq rate limits are hit.

Known Limitations

Limitation Impact on SimuOrg Mitigation Applied
Context window: 8,192 tokens Cannot process large RAG context or full employee lists RAG limited to top-3 scenario chunks; employee data never sent to LLM
JSON hallucination (8B Fallback) 8B fallback may occasionally misformat complex JSON Regex JSON block extractor + Pydantic validation + retry loop
Out-of-range numeric values Simulation receives invalid policy parameters Hard bounds enforcement post-generation; values clamped to safe ranges
Groq rate-limits 429 errors during burst usage Automatic transparent failover to local 8B Ollama instance

8. Performance Metrics & Model Evaluation

SimuOrg tracks model quality across five dimensions: discriminative power, calibration accuracy, simulation fidelity, classification effectiveness, and explainability coverage.

8.1 Discriminative Power — AUC-ROC

AUC-ROC measures how well the model ranks employees by their true attrition probability, independent of any threshold. A score of 1.0 means perfect separation; 0.5 means random. SimuOrg targets ≥0.80, which is considered clinically actionable in attrition research.

Dataset Type AUC-ROC Achieved Interpretation
IBM HR Analytics (clean, 1,470 rows) 0.83–0.87 Strong signal — reliable attrition ranking
Synthetic dataset (balanced 50/50) 0.88–0.92 Near-ideal conditions — not representative of production
Low-quality dataset (quality score < 60) 0.62–0.71 Marginal — model will operate with warning flag

8.2 Probability Calibration — Expected Calibration Error (ECE)

Calibration measures whether the model's predicted probabilities match observed frequencies. If the model says "30% chance of leaving," 30% of those employees should actually leave. ECE is the average gap between predicted confidence and empirical accuracy across 10 probability bins.

Calibration Stage ECE Notes
Raw XGBoost (uncalibrated) 0.12–0.18 Over-confident — probabilities cluster near 0 or 1
After IsotonicRegression 0.03–0.05 Well-calibrated — probabilities reflect true frequencies
After binary-search alpha scaling <0.03 Simulation-aligned — sim attrition matches history ±0.5%

8.3 Classification Effectiveness — Precision, Recall, F1

At the F1-optimal threshold, the model balances false positives (unnecessary HR interventions) and false negatives (missed at-risk employees). Recall is weighted higher in HR contexts because missing a flight-risk employee is costlier than a false alarm.

Metric Target Achieved Business Meaning
Precision ≥0.65 0.68–0.76 Of flagged employees, this fraction truly leaves — HR intervention accuracy
Recall ≥0.60 0.62–0.71 Of employees who leave, this fraction was caught early — coverage rate
F1 Score Maximised 0.65–0.73 Harmonic mean — primary optimisation target for threshold selection
Training Time ≤60s 15–45s For 1,000–5,000 row datasets on standard hardware

8.4 Simulation Fidelity

Unlike pure ML metrics, simulation fidelity measures whether the agent-based engine produces realistic organisational behaviour over time.

Metric Value Notes
Calibration tolerance ±0.5% vs historical rate Achieved via binary-search alpha loop
Monte Carlo runs (default) 50 runs per simulation Provides stable confidence intervals; configurable
Tracked KPIs per month 5 (attrition, headcount, stress, motivation, productivity) Stored as monthly JSON in SimulationResult

12. Lessons Learned

12.1 Engineering

  • Calibrate before simulating. Raw ML probabilities are systematically miscalibrated. Without the empirical binary-search loop, simulation results were unrealistic regardless of model AUC-ROC.
  • BackgroundTasks are not workers. FastAPI BackgroundTask runs in the same process. CPU-heavy ML training at scale requires a proper task queue (Celery, ARQ) to avoid event loop blocking.
  • Schema normalisation is the hardest part of ingestion. Mapping 200+ column name variants reliably is where most edge-case engineering time was spent.
  • Session isolation must be designed in from day one. Retrofitting UUID-scoped DB queries after the fact would require a complete rewrite of the data access layer.

12.2 Machine Learning

  • AUC-ROC alone is insufficient. A model can score 0.85 AUC-ROC on imbalanced data while being useless in practice. Threshold tuning and calibration are as critical as base model quality.
  • SHAP TreeExplainer is the correct choice for XGBoost. Exact, fast, and tree-aware. KernelExplainer would be 100x slower with no accuracy benefit for gradient-boosted trees.
  • Domain knowledge beats raw feature count. Five derived interaction features (income_ratio, overtime_stress, etc.) contributed more to AUC-ROC than adding 20 additional raw columns.

12.3 Product

  • Natural language interfaces lower the barrier to insight. Allowing a CEO to type a policy in plain English dramatically increases the system's perceived accessibility.
  • Showing uncertainty builds trust. 95% confidence bands on Monte Carlo forecasts make stakeholders more, not less, confident in the model's outputs.
  • Explainability is a business requirement. SHAP attributions were the most consistently requested feature by HR professionals during feedback sessions.

  • Real-Time HR Integration. Webhooks from Workday, BambooHR, and SAP to continuously update employee state without manual CSV uploads.
  • Reinforcement Learning Policy Optimiser. RL agent that searches the policy space to recommend the intervention sequence minimising attrition over a target horizon.

14. Conclusion

SimuOrg is a complete, production-grade demonstration of how modern AI techniques — gradient-boosted ML, agent-based simulation, LLM orchestration, and RAG — can be composed into a single coherent product that delivers measurable business value.

Built from first principles over twelve weeks, the system progressed from a bare FastAPI skeleton to a fully containerised, cloud-deployable workforce intelligence platform. Every major design decision — from isotonic calibration to dual-LLM fallback to session-scoped multi-tenancy — was made to prioritise reliability, explainability, and real-world usability over academic novelty.

SimuOrg proves that an individual developer, with the right toolchain and disciplined engineering, can build enterprise-grade predictive HR analytics infrastructure at near-zero marginal cost.

Three core innovations define the system: (1) empirical calibration closing the gap between ML probability outputs and simulation reality; (2) natural-language policy input removing the technical barrier for C-suite users; and (3) per-employee SHAP attribution making recommendations auditable and trustworthy to HR professionals.
